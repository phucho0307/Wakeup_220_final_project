# WakeUp: A Math-Puzzle Alarm Clock (A Study in API-First Architecture)

Welcome to the WakeUp demonstration. This project is designed to show the full lifecycle of a modern web application, from architectural negotiation to automated implementation.

## 1. Project Philosophy

Instead of jumping straight into code, this project uses an API-First approach. We define the "Contract" (the rules of engagement) before building the "Engine" (the server) or the "Interface" (the client). This allows different teams, or AI agents, to work in parallel without breaking the system.

## 2. What WakeUp Does

WakeUp is an alarm clock that does not let you sleep through it. A user sets a countdown timer from the client. When the countdown reaches zero, the alarm starts ringing: both the server machine and the client browser produce an alarm sound. To make the ringing stop, the user must correctly solve a math problem generated by the server. The problem is a chain of two operations (addition and subtraction only) on 2 to 3 digit integers. A wrong answer keeps the alarm ringing with the same problem still showing. Only a correct answer dismisses the alarm.

The goal is simple: force the brain to wake up before the noise turns off.

## 3. Repository Structure

This is a monorepo organized by Responsibility:

- wakeup/
  - design/ : THE ARCHITECT'S OFFICE. Contains wakeup-api.md (the law), server-design.md (the engine blueprint), and client-design.md (the UI blueprint). When the coding agent completes a task, it adds a new entry to server-changes.md or client-changes.md, which can then be used to refactor the design docs later.
  - server/ : THE ENGINE. The Spring Boot backend that manages alarm state, the countdown deadline, math problem generation, and answer checking.
    - server/config/ : THE CONFIG DATA. Contains alarm-defaults.txt, the text file that defines default alarm behavior such as digit range and ring volume.
  - client/ : THE INTERFACE. The React frontend that renders the countdown, the ringing screen, and the math input.
  - CLAUDE.md : THE AI GUARDRAILS. The specific instructions used to guide the Claude CLI.

## 4. How the Pieces Fit Together

1. The Config (server/config/alarm-defaults.txt): You can change default alarm behavior by editing this text file. No code changes required.
2. The Server (server/): Stores the alarm deadline, decides when the alarm should ring, generates the math problem, and checks answers. It does not care if the client is a website, a mobile app, or a CLI.
3. The Client (client/): Asks the server "what is the alarm state?" on a short interval and reacts to whatever it is told. It is "dumb": it never decides on its own that time is up and never checks the math answer itself.
4. The API (design/wakeup-api.md): This is the glue. As long as both the server and client follow this document, the system works.

## 5. Learning Objectives

As you explore this project, pay attention to:

- Separation of Concerns: Notice how the alarm logic and the answer checking live in the server, while the visual logic lives in the client.
- Single Source of Truth: The server owns the countdown deadline. The client only polls and displays. A page refresh never loses the alarm.
- Statelessness of Identity: The server does not "know" who you are until you send your username as a parameter.
- Documentation as Code: See how the .md files in the design/ folder are not just notes, they are the blueprints that generated the application.
